## 1. Claude Code 習熟度レベル（5段階）

Claude Code には、利用者の習熟度に応じた5段階の最適化レベルがある。レベルが上がるほど「人間が繰り返し指示していた作業」がシステムに吸収され、トークンコストが下がり品質が安定する。

```
Level 1: 素のプロンプト
Level 2: + CLAUDE.md（プロジェクトルールの自動読み込み）
Level 3: + Skills（手順書のオンデマンド注入）
Level 4: + Hooks（自動フォーマット・品質チェック）
Level 5: + Agents（並行レビューの自動実行）
```

### レベル別の概要

| レベル | 追加要素 | 人間の操作 | AIの動作 |
|---|---|---|---|
| **1** | なし | 毎回ルールをプロンプトで指示 | 指示された内容のみ実行 |
| **2** | CLAUDE.md | セッション開始時に自動読み込み | プロジェクトルールを常に意識 |
| **3** | Skills | `/fix-issue` 等のコマンドで手順を注入 | 詳細手順に従い網羅的に実行 |
| **4** | Hooks | ファイル保存時・セッション終了時に自動実行 | フォーマット・テスト・解析を自動実行 |
| **5** | Agents | 指示に応じて並行プロセスで起動 | セキュリティ・パフォーマンス・ラベル漏れを並行チェック |

### レベル移行の判断基準

| 状況 | 推奨レベル |
|---|---|
| Claude Code を試しに使ってみる | Level 1 |
| プロジェクトのルールを毎回伝えるのが面倒 | Level 2 |
| 同じ作業手順を何度も説明している | Level 3 |
| `flutter analyze` 等を毎回手動で頼んでいる | Level 4 |
| セキュリティ・パフォーマンスレビューを毎回依頼している | Level 5 |

## 2. 各レベルの構成要素と設定方法

### Level 2: CLAUDE.md

プロジェクトルートに `CLAUDE.md` を配置する。セッション開始時に自動で読み込まれる。

**設計原則: 150行以内に抑える。** 詳細手順は Skills に移行し、ここにはルールの要約のみ残す。

```markdown
# プロジェクト名 - Claude Code 運用ガイド

## プロジェクト概要
- 技術スタック: Flutter / Dart / Drift / Riverpod
- デプロイ先: GitHub Pages

## コミットルール
- テストコードの追加・修正を伴わないソースコード変更はコミットしない

## コミット前チェック（毎回必須）
1. 横展開チェック
2. セキュリティチェック
3. パフォーマンスチェック
4. デプロイチェック（flutter analyze → flutter test → flutter build web）
5. 単体テスト
```

**やりがちな失敗:** CLAUDE.md に全手順を書き込むとファイルサイズが膨らみ（7,000バイト超）、毎セッションのトークンコストが増大する。本プロジェクトでは Level 2 時代に一部セッションでコストが 26%増加した。

### Level 3: Skills

`.claude/skills/` ディレクトリにMarkdownファイルを配置する。`/コマンド名` で呼び出した時だけトークンを消費する。

**本プロジェクトの Skills 一覧:**

| ファイル | コマンド | 用途 |
|---|---|---|
| `fix-issue.md` | `/fix-issue` | SonarCloud/UIテストレポートから問題を修正 |
| `release.md` | `/release` | バージョンアップ・リリース作業 |
| `check-deploy.md` | `/check-deploy` | GitHub Actions デプロイ失敗の調査・修正 |
| `update-labels.md` | `/update-labels` | ラベル・メッセージの変更と横展開 |

**Skill ファイルの構造:**

```markdown
---
name: fix-issue
description: 問題修正スキル
---

# 手順

1. レポートファイルを読む
2. ソースコードを修正
3. テストを追加・修正
4. チェックリストを実行
```

**設計原則:** 「2回以上同じ手順を口頭で伝えた」ら Skill 化する。

### Level 4: Hooks

`.claude/settings.json` の `hooks` セクションで定義する。人手を介さず自動実行される。

**本プロジェクトの Hooks:**

```jsonc
{
  "hooks": {
    "PostToolUse": [
      {
        // ファイル編集後に自動フォーマット
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "dart format --fix $CLAUDE_FILE_PATH"
        }]
      }
    ],
    "Stop": [
      {
        // セッション終了時の品質ゲート（コマンド）
        "hooks": [{
          "type": "command",
          "command": "flutter analyze && flutter test && grep検出..."
        }]
      },
      {
        // セッション終了時の品質ゲート（AI判定）
        "hooks": [{
          "type": "prompt",
          "prompt": "横展開・セキュリティ・パフォーマンス・テスト整合性をチェック..."
        }]
      }
    ]
  }
}
```

**Hook の種類と使い分け:**

| 種類 | トリガー | 用途例 |
|---|---|---|
| `PostToolUse` (command) | ファイル編集後 | 自動フォーマット |
| `Stop` (command) | セッション終了時 | `flutter analyze`、`flutter test`、禁止語検出 |
| `Stop` (prompt) | セッション終了時 | AI による横展開チェック、セキュリティチェック |

**設計原則:** 「毎回手動で依頼しているチェック」があれば Hook 化する。command は確実に実行され、prompt は AI の判断力を活用する。

### Level 5: Agents

`.claude/agents/` ディレクトリに Markdown ファイルを配置する。独立したサブプロセスとして並行実行できる。

**本プロジェクトの Agents:**

| ファイル | トリガー例 | チェック内容 |
|---|---|---|
| `security-reviewer.md` | 「セキュリティチェックして」 | XSS、SQLインジェクション、入力バリデーション、機密情報 |
| `performance-reviewer.md` | 「パフォーマンスチェックして」 | N+1クエリ、Paint生成、Provider設計、非同期並列化 |
| `label-checker.md` | 「ハードコードの文字列チェック」 | `AppLabels` 未適用の日本語文字列検出 |

**Agent ファイルの構造:**

```markdown
---
name: security-reviewer
description: セキュリティ観点でコードをレビューする
tools: [Read, Glob, Grep, Bash]
---

# チェック項目

1. XSS: ユーザー入力のサニタイズ確認
2. SQLインジェクション: パラメータ化クエリの使用確認
...
```

**設計原則:** Agent は「メインの作業コンテキストに影響しないレビュー作業」に使う。コードの修正はメインセッションで行い、Agent はチェック・報告に専念させる。

## 3. トークンコスト最適化の実測値

本プロジェクトの実測値に基づく、レベル別のコスト比較。

### セッション種別ごとの比較

| セッション種別 | Level 1 | Level 2 | Level 5 | Level 5 の削減率 |
|---|---|---|---|---|
| 問題修正 | 6,000 tokens | 5,800 | 2,400 | **-60%** |
| リリース作業 | 3,600 tokens | 4,520 | 800 | **-78%** |
| ラベル変更 | 3,300 tokens | 3,040 | 700 | **-79%** |

### 月間推定（60セッション/月）

| レベル | 月間トークン | Level 1 比 |
|---|---|---|
| Level 1 | 269,400 | 基準 |
| Level 2 | 282,800 | +5%（CLAUDE.md 肥大化） |
| Level 3 | 116,200 | **-57%** |
| Level 4 | 102,800 | **-62%** |
| Level 5 | 90,000 | **-67%** |

### 固定コンテキスト（毎セッション読み込み）

| レベル | 固定バイト数 | 内訳 |
|---|---|---|
| Level 2 | 16,574 bytes | CLAUDE.md に全手順を詰め込んだ状態 |
| Level 5 | 5,161 bytes | CLAUDE.md を要約のみに圧縮 |
| 削減率 | **-69%** | |

## 4. 品質自動化の到達レベル

### 自動化マトリクス

| 品質チェック | Level 1-2 | Level 3 | Level 4-5 |
|---|---|---|---|
| コードフォーマット | 手動依頼 | 手動依頼 | **自動**（PostToolUse Hook） |
| 静的解析 | 手動依頼 | Skill内に手順あり | **自動**（Stop Hook command） |
| テスト実行 | 手動依頼 | Skill内に手順あり | **自動**（Stop Hook command） |
| 禁止語検出 | 気づいた時に依頼 | Skill内に手順あり | **自動**（Stop Hook command） |
| 横展開チェック | 気づいた時に依頼 | Skill内に手順あり | **自動**（Stop Hook prompt） |
| セキュリティチェック | 気づいた時に依頼 | 手動依頼 | **自動**（Stop Hook prompt + Agent） |
| パフォーマンスチェック | 気づいた時に依頼 | 手動依頼 | **自動**（Stop Hook prompt + Agent） |

### 品質ゲートの実行タイミング

```
ファイル編集 → [PostToolUse] dart format 自動実行
    ↓
開発作業の継続
    ↓
セッション終了 → [Stop command] flutter analyze → flutter test → 禁止語チェック
    ↓
              → [Stop prompt] AI が横展開・セキュリティ・パフォーマンスを最終チェック
    ↓
問題があれば Claude Code がセッション内で自動修正
```

## 5. AI駆動開発の設計原則

### 5.1 CLAUDE.md は「ルールブック」であり「手順書」ではない

| NG | OK |
|---|---|
| 全手順を CLAUDE.md に記載（7,000バイト超） | ルールの要約のみ（150行以内） |
| 詳細な grep コマンド例を列挙 | 「横展開チェック」の一言 + 詳細は Skill へ |

### 5.2 「2回言ったら自動化する」

| 回数 | アクション |
|---|---|
| 1回目 | 口頭（プロンプト）で指示 |
| 2回目 | Skill 化 or Hook 化を検討 |
| 毎回 | Hook（自動実行）に昇格 |

### 5.3 人間とAIの役割分担

| 役割 | 人間 | AI |
|---|---|---|
| **何を作るか** | プロダクトビジョン、ドメイン設計 | - |
| **どう作るか** | 最終的なUX判断 | アーキテクチャ提案、技術選定、実装 |
| **品質保証** | 実機テスト、視認性確認 | テスト生成、静的解析、セキュリティレビュー |
| **リリース判断** | コミット・プッシュの実行 | チェックリスト自動実行、ブロッカー検出 |

### 5.4 AIへの指示の粒度

| 粒度 | 例 | 結果 |
|---|---|---|
| 抽象的すぎる | 「いい感じにして」 | AIが迷走、手戻りが発生 |
| **適切** | **「サブスク解約後に体験版に戻る仕組みを作って」** | **AIが設計提案→実装→テスト** |
| 具体的すぎる | 「stripe_service.dart の 45行目に if文を追加して」 | AIの設計力を活かせない |

**最適な粒度:** 「何を実現したいか」を伝え、「どう実装するか」はAIに任せる。

### 5.5 レビューの仕方

AIが生成したコードを「全行読む」のではなく、**以下の観点でレビュー**する:

1. **意図通りか** — 要件を満たしているか（動作確認）
2. **副作用がないか** — 既存機能が壊れていないか（テスト結果）
3. **設計が妥当か** — 過剰な抽象化や不要な複雑性がないか（差分の俯瞰）

## 6. 実践パターン集

### パターン1: 未経験技術での開発

```
人間: 「Flutter で Drift ORM を使ったタスク管理画面を作って」
  ↓
AI: アーキテクチャ設計 → テーブル定義 → DAO → Service → Provider → Page → Test
  ↓
人間: 実機で動作確認 → UI調整を指示
```

**効果:** Flutter 未経験でも初日からプロダクションコードを生成できる。学習コスト = 0。

### パターン2: 大規模リファクタリング

```
人間: 「全てのハードコード文字列を AppLabels に外部化して」
  ↓
AI: 478件のハードコードを検出 → 550定数を生成 → 全ファイル置換 → テスト修正
  ↓
Hook: 禁止語チェックで残留を自動検出
```

**効果:** 人間なら1週間以上かかる作業を1セッションで完了。Hook により漏れを自動検出。

### パターン3: 外部サービス連携

```
人間: 「ユーザーが自分でサブスクを解約できるようにして」
  ↓
AI: Stripe Customer Portal の設計提案 → Flutter側実装 → Apps Script側の仕様提示
  ↓
人間: Apps Script にコードをデプロイ → 動作確認
  ↓
AI: CORS問題を検出 → エラーハンドリング修正 → 3層検証の追加提案
```

**効果:** AIが設計→実装→問題検出→修正のサイクルを高速に回す。人間は意思決定とデプロイに集中。

### パターン4: バグ修正の横展開

```
人間: 「書籍編集でメモが保存されない」
  ↓
AI: デバッグログ追加 → 原因特定（ダイアログのスクロール問題）→ 修正
  ↓
AI: 同様のパターンが他のダイアログにないか自動チェック → SnackBar の視認性問題も検出
  ↓
Hook: 横展開チェックで旧パターンの残留を自動検出
```

**効果:** 1つのバグ修正から関連問題を芋づる式に発見・修正。

## 7. アンチパターン集

### アンチパターン1: CLAUDE.md の肥大化

**症状:** CLAUDE.md に全手順を詰め込み、毎セッション 7,000バイト以上を読み込む。

**問題:** トークンコストが 26%増加。リリース作業のような簡単なセッションでも無関係なルールを読み込む。

**対策:** CLAUDE.md は 150行以内。詳細手順は Skills に移行。

### アンチパターン2: Hook の過剰設定

**症状:** 全てのチェックを Hook に入れ、セッション終了時に5分以上のチェックが走る。

**問題:** 開発の流れが止まる。軽微な修正でもフルチェックが走る。

**対策:** Hook は「絶対に漏らしてはいけないチェック」のみ。頻度の低いチェックは Agent として手動起動。

### アンチパターン3: AIに全てを任せる

**症状:** 「いい感じにリファクタリングして」「改善して」と抽象的に指示。

**問題:** AIが不要な変更を加える。意図しないファイルが修正される。

**対策:** 「何を」「なぜ」変えたいかを明確に伝える。「どう」はAIに任せてよい。

### アンチパターン4: AIの出力を検証しない

**症状:** AIが生成したコードをそのままコミット。

**問題:** 実機でのUI崩れ、外部サービス連携の不具合を見逃す。

**対策:** コミット前にローカルで動作確認。UIの視認性・操作性は人間が必ず確認する。

## 8. 他プロジェクトへの展開方法

### テンプレートリポジトリ

本プロジェクトの Level 5 構成をテンプレート化したリポジトリを用意している。

**リポジトリ:** `ClaudeCodeTemplate`

### 導入手順

```bash
# 1. テンプレートをクローン
git clone https://github.com/your-org/ClaudeCodeTemplate my-project/.claude-setup

# 2. セットアップスクリプトを実行（プレースホルダーを置換）
cd my-project/.claude-setup
bash setup.sh

# 3. 生成されたファイルをプロジェクトに配置
cp -r output/* ../
```

### プレースホルダー

| プレースホルダー | 説明 | 例 |
|---|---|---|
| `{{PROJECT_NAME}}` | プロジェクト名 | ユメログ |
| `{{TECH_STACK}}` | 技術スタック | Flutter / Dart / Drift / Riverpod |
| `{{TEST_COMMAND}}` | テスト実行コマンド | `flutter test` |
| `{{ANALYZE_COMMAND}}` | 静的解析コマンド | `flutter analyze` |
| `{{BUILD_COMMAND}}` | ビルドコマンド | `flutter build web` |
| `{{FORMAT_COMMAND}}` | フォーマットコマンド | `dart format --fix` |
| `{{PROJECT_DIR}}` | プロジェクトディレクトリ | `lib/` |

### 段階的導入の推奨

```
Week 1: Level 2（CLAUDE.md を配置）
  ↓ 2回以上同じ手順を伝えたら
Week 2: Level 3（Skills を追加）
  ↓ 毎回手動で依頼しているチェックがあれば
Week 3: Level 4（Hooks を追加）
  ↓ レビュー作業を並行化したければ
Week 4: Level 5（Agents を追加）
```

**一気に Level 5 を目指さない。** プロジェクトの課題が見えてから、その課題を解決する仕組みとして各レベルの要素を追加する。
